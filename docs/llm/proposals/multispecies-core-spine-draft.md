# Multi-species CORE SPINE — verified plan (DRAFT)

> **`multispecies-core-spine-design` workflow (10 agents, adversarially verified). DESIGN ONLY. Phase 1 of: core logic → ecosystem → UI.**

## Summary

VERIFIED multi-species CORE-SPINE plan for gene-sim, sequenced as gated slices. The single-species core encodes "species = one genome" as three per-run singletons: GenomeRes(Genome) (lib.rs:86), BaseGrowthRate(f64) (lib.rs:92), and the Simulation.gp_map field (lib.rs:385), with selection() (lib.rs:249-319) a constant-N Wright-Fisher pool over ALL orgs and hash_world() (lib.rs:792-846) pinned at 0xf795_eac4_112f_acd5 (lib.rs:885). The spine turns those into an ordered SpeciesId-indexed SpeciesRegistry, tags every organism with a heritable off-stream Species(SpeciesId), and (later) runs selection as S independent constant-size Wright-Fisher pools. Sequencing (each a separate commit/gate): R3-A registry+tag (HASH-NEUTRAL, ADDITIVE — no sign-off, the literal is the acceptance proof) → R3-B per-species Wright-Fisher (DELIBERATE re-pin, human-signed) → F3 resource coupling (separate deliberate re-pin, human-signed). R3-A is the first slice and is provably byte-identical for the 1-species default. The adversarial verdict on R3-A is "holds" (additive variant only); R3-B is "uncertain" (the first disjunct "N=1 still hash-identical" is FALSE and is dropped — R3-B survives only as a scoped+ledgered re-pin); the coupling design is "holds". All must_fix items are folded in below.


## Registry spine

SpeciesRegistry is an ORDERED Vec indexed by a contiguous SpeciesId(u16) newtype — NEVER a HashMap iterated in sim logic (inv #3), mirroring the genome crate's LocusId Vec+newtype pattern. Each entry is SpeciesEntry{ name:String, genome:Genome, gp_map:gp::OntologyMap, base_growth:f64, target_pop:u32 }, replacing the three singletons (GenomeRes lib.rs:86, BaseGrowthRate lib.rs:92, Simulation.gp_map lib.rs:385). base_growth = gp_map.express(&genome).get(Trait::GrowthRate).unwrap_or(0.5) computed once per entry at reset, exactly mirroring lib.rs:438-439; gp_map carries the per-species TraitMap via gp::trait_map_for(key) (gp.rs:225) — gp.rs needs NO change (it is already per-species-ready). target_pop = that species' initial member count (the per-species ADR-005 constant-population target for R3-B). N=2 entry registry held as a Vec<SpeciesEntry> in ascending SpeciesId order. RESET BUILDS IT from a filesystem-free roster: reset takes an already-built Vec<RosterEntry{name,genome,gp_map,entity_count}> (the harness does file I/O + trait_map_for, core stays I/O-free per inv #2); species_roster_or_default() returns, with no manifest, exactly vec![RosterEntry{ name:"default", genome:genome::sample_genome(), gp_map:gp::OntologyMap::new(gp::default_plant_trait_map()), entity_count:config.entity_count }]. ONE-SPECIES DEFAULT STAYS BYTE-IDENTICAL: that single entry reproduces base_growth=0.6 (pinned gp.rs:283 / f2_default_plant_map_pins_expression), the same spawn draw stream, and (R3-A) the same hashed singletons — proven by the unmoved literal. RECOMMENDED ENTRY POINT: add reset_with_roster(config, env, Vec<RosterEntry>) and have the existing reset_with_genome_and_map delegate with a 1-entry roster, so there is ONE spawn/registry code path (the single-path discipline at lib.rs:768-788).


## Organism SpeciesId

Add a heritable ECS Component Species(SpeciesId) near Position (lib.rs:152). It is assigned at SPAWN from the roster loop index / current SpeciesId — NEVER from rng.next_u64() — so it adds ZERO SimRng draws, exactly the Position / PLACEMENT_STREAM_BASE off-stream precedent (lib.rs:167,209,462). This is the single load-bearing implementation constraint (must_fix): any extra next_u64 shifts the entire downstream stream (selection's 2N draws + the trailing final_word at lib.rs:826) and breaks the literal even at N=1. The spawn loop MUST stay `for i in 0..entity_count { OrgId(i), ... }` with global contiguous OrgIds — do NOT make OrgId per-entry-relative or reorder spawning, because hash_world keys/sorts/folds by OrgId (lib.rs:821,835). Species is appended to the spawn tuple (lib.rs:454-463) after placement. Inheritance (R3-B): the tag rides the offspring tuple alongside genotype/drought/thermal/position (lib.rs:270-273, 304) — offspring inherit the parent's Species, lineages never cross-breed or re-merge. The 4-draw spawn order (g0,energy,drought,thermal — lib.rs:449-452) is untouched.


## Per-species selection (R3-B re-pin)

R3-B (DELIBERATE re-pin): replace the single pool with S INDEPENDENT FIXED-SIZE Wright-Fisher pools iterated in ASCENDING SpeciesId order (per-species ADR-005 constant population — INDEPENDENT pools, NOT a shared carrying capacity; shared-carrying-capacity competition is the energy/substrate-funded CHEMOSTAT F3 phase, deferred, to keep this re-pin minimal). DRAW ORDER, fully specified: snapshot parents as (species,id,g,d,t,x,y) via q.iter() then `parents.sort_unstable_by_key(|p| (p.species, p.id))`. For sid in 0..registry.entries.len() ASCENDING: take the contiguous (species,id)-sorted slice for that species; build that species' own cumulative weight table using fitness(entries[sid].base_growth, g) × soil_factor × clim_factor (the existing strictly-positive [0.5,1.5] products, lib.rs:286-288); emit exactly entries[sid].target_pop offspring with EXACTLY 2 draws each in fixed order — select-parent (lib.rs:298) then disperse (lib.rs:301). Per-generation draw COUNT is preserved at 2N total (Σ over pools of 2·target_pop). WRITE-BACK (must_fix C): each pool zips its target_pop offspring against ITS OWN (species,id)-sorted parent-id slice, merged into the single OrgId-keyed BTreeMap (lib.rs:308) — OrgIds are globally unique so no cross-species collision; do NOT use a global parents.iter().zip(offspring) over all species (the current lib.rs:308-309 form would mis-pair across pools). EDGE SEMANTICS (must_fix, decide before R3-B): a pool with <2 live members draws ZERO (the existing early-return fast path, lib.rs:274); target_pop is the regeneration target only while ≥2 members exist. FLOAT NOTE (must_fix D): per-pool `total` is a left-fold over only that pool's members, so for N≥2 the partial sums differ from the single global total — this IS the intended re-pin and is reproducible only under inv #3's same-build/same-platform clause; all weight math stays multiply/add (no transcendentals). RNG = the single explicitly-threaded ChaCha8Rng (lib.rs:78), single-threaded .chain() (ADR-002), no HashMap iteration.


## Resource coupling seam (F3 phase-2)

The species interact through the F1 ResourceField (resource.rs), today inserted UNWIRED + #[allow(dead_code)] at lib.rs:110/480 — hash-neutral, proven by the unmoved literal (the F1/climate/Ledger insert-unwired precedent). This is PHASE-2 (the F3 ECOSYSTEM coupling), NOT the spine, and is a SEPARATE deliberate re-pin from R3-B (sequence them independently so each re-pin is minimal/bisectable). The seam: (1) a new RNG-FREE nutrient_dynamics system runs AFTER metabolism, BEFORE selection — schedule becomes (advance_tick, metabolism, nutrient_dynamics, selection).chain() (lib.rs:489). It mutates ResourceFieldRes free_nutrient/detritus per tick as f(nearby organisms of ALL species) via a per-species TrophicRole (Autotroph/Decomposer): plants deplete free_nutrient + shed detritus; the decomposer mineralizes detritus→free_nutrient — the obligate trophic loop (plant ⊣ detritus ⊢ decomposer) that makes species genuinely interact through the substrate rather than react to a frozen field. (2) A ResourceModifier trait (twin of soil::EnvironmentModifier soil.rs:175, inv #5 pluggable) returns a strictly-positive [0.5,1.5] RNG-free factor multiplied into the selection weight product (lib.rs:286-288) — perturbs fitness VALUES not draw COUNT. MUST_FIX for this phase: nutrient_dynamics accumulates via COMMUTATIVE integer sums then applies per-cell row-major clamped transfers (min(demand,pool)) — never per-organism sequential depletion over the archetype-order Query (the lib.rs:228 hazard generalized); migrate free_nutrient/detritus to i64 J-quanta; assert ledger::Ledger.closes() each tick (a conservation guard stronger than the bit-hash); fold the dynamic pools row-major into hash_world and MOVE the literal in the same commit. Do NOT make the field dynamic in the spine slice; do NOT per-species the shared substrate resources (SoilFieldRes/ClimateFieldRes/ResourceFieldRes/Ledger inserts lib.rs:470-484 stay shared in R3).


## Per-species observe / snapshot

Per-species observe + snapshot are PURE projections (no next_u64, no mutation — the observe/snapshot contract at lib.rs:512/544), so they are HASH-NEUTRAL by construction and ship in the R3-A spine slice. observe: Observation (lib.rs:352) gains per_species: Vec<SpeciesObservation{ species_id:u16, name:String, population_size:u32, allele_freq:f64, phenotype:Phenotype }> indexed by ascending SpeciesId (Vec, never HashMap). mean_genotype/allele_freq become per-species by partitioning the OrgId-sorted rows by the Species tag in ascending SpeciesId order (no HashMap iteration); the existing TOP-LEVEL allele_freq/population_size stay as the global aggregate (byte-identical for 1 species, no caller breaks). snapshot: GridSnapshot (snapshot.rs:54) gains species: Vec<SpeciesPlane{ species_id:u16, density:Vec<f32>, allele_freq:Vec<f32> }> (each width×height row-major, ascending SpeciesId), binned in the existing OrgId-sorted loop. The binary format (snapshot.rs:84 write_snapshot_bytes, CHANNEL_COUNT=6 snapshot.rs:41) is EXTENDED ADDITIVELY: keep the 6 fixed channels as the all-species aggregate, then append species_count + per-species (id, density-plane, allele-plane) as a trailer so old readers parse the prefix. Godot reads it read-only (inv #2 — no biology in GDScript). These feed the future per-species UI without touching the hash.

## Invariant risks
- INV #3 (determinism, default byte-identical): the ENTIRE R3-A claim rests on (a) Species assigned off-stream — any rng.next_u64() for the tag breaks the literal; (b) the additive variant — keeping GenomeRes/BaseGrowthRate/gp_map so hash_world is literally unmodified; (c) global contiguous OrgId(i) spawning unchanged. A collapse-in-R3-A would edit hash_world and must be byte-verified, not assumed.
- INV #3 (R3-B re-pin discipline): R3-B is NOT hash-identical at N=1 (Species + species_count + param-sum enter hash_world). The re-pin must be minimal (preserve 2N draw count + per-pool internal order), human-signed, literal moved in the SAME commit with history-comment + ledger line. Risk: writing an N=1 test asserting the OLD literal after R3-B (must not).
- INV #3 (no HashMap / ordered draws): registry is a Vec; per-pool grouping iterates 0..len ascending over a (species,id)-sorted slice; write-back is the OrgId BTreeMap. Per-pool write-back must zip each pool's offspring against ITS OWN parent slice (not a global zip).
- INV #3 (no transcendentals / float reassociation): per-pool `total` left-folds differ from the global total at N≥2 — intended, reproducible only under same-build/same-platform; keep all weight math multiply/add.
- INV #6 (granularity): edits address SpeciesId + Region disc (MIN_REGION_RADIUS), never an organism handle; per-species pools operate per species, orgs stay ECS entities. The Species tag is a Component, not an RL agent.
- INV #2 (biology in core): registry/gp_map/express/TrophicRole live ONLY in sim-core; godot consumes per-species snapshot planes read-only.
- INV #5 (pluggable): the future ResourceModifier mirrors soil::EnvironmentModifier; keep any new inter-species coefficient ZERO/identity at introduction so wiring is hash-neutral and only activation re-pins.
- R3-B + F3 are BOTH stop-the-line re-pins requiring human sign-off; do NOT auto-proceed past R3-A into either.

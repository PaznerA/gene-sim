# ADR-014 — Phase Rel: inter-species relations (mutualism / parasitism / predation) via an in-core community matrix + a process-boundary vector-DB index for the relations view (DRAFT, depends on R3/ADR-013, 🛑 human sign-off)

> **DRAFT — generated by the `rel-relations-vectordb-design` workflow (scheduled run). DESIGN/RESEARCH PROPOSAL ONLY — awaiting human sign-off. Depends on R3/ADR-013. No code written.**

## Context

Phase Rel adds (a) a new READ-ONLY relations view of inter-species / inter-lineage relations — mutualism/synergy, parasitism, predation — and (b) a vector-DB index over per-species genome/phenotype embeddings for similarity + lineage-neighbourhood ("nearest cousins") discovery. It conceptually depends on R3 multi-species (ADR-013, still in design): a relation is meaningless single-species, so this ADR is proposed to land AFTER R3 and is modeled abstractly over an opaque GroupId/SpeciesId so it slots onto whatever R3 lands.

Two invariants dominate the design:
- inv #1 (process boundary): a vector DB must NOT link into the deterministic game binary or sim-core. The enforced template is crates/oracle-slim (std-only, shells out via std::process::Command, no GPL/DB crate). scripts/check_license.sh has TWO assertions: (A) a GPL-OR-aware scan over the whole resolved tree, and (B) a belt-and-suspenders zero-normal/build-deps check HARD-KEYED to the crate literally named "oracle-slim" (lines 56-68). Verified in-repo.
- inv #3 (determinism): a vector DB is approximate/non-deterministic by nature (HNSW/ANN ordering is insertion-order/threading/float-sensitive; inv #7 version-pinning does NOT make it bit-stable). Anything the DB produces must therefore stay OFF the bit-deterministic sim hash, or cross back only through a deterministic, ordered, integer-quantized channel.

Existing seams to reuse (verified): soil::EnvironmentModifier (soil.rs:174) and climate::ClimateModifier (climate.rs:99) both return a strictly-positive multiplicative fitness_factor in a bounded [0.5,1.5] band (preserves ADR-005 no-extinction). selection() (lib.rs:218) multiplies these factors into a Wright-Fisher weight in stable OrgId order, with ordered-BTreeMap write-back and no HashMap iteration. climate's TemperatureMatchModifier scales pressure by climate EXTREMITY so the neutral/pinned world stays selection-neutral — the precedent for keeping a new coupling hash-neutral until deliberately driven. gp.rs Trait::ALL is a fixed [Trait;5], express() is HashMap-free and clamps to [0,1] — the natural quantizable embedding. The last deliberate determinism re-pin was ADR-012 E3 (ThermalTol → 0x9fad_2c9f_d298_f73a).

Two designs were evaluated. Design 1 "Sidecar Relations Index" (file-based sqlite-vec/LanceDB written by the harness, DB powers the VIEW only, dynamics are an exact in-core quantized-cosine community matrix; risk: low). Design 2 "Lineage Web" (long-lived Qdrant server, ANN neighbourhood results re-quantized back into selection through a firewall; risk: high). Both read the codebase accurately; both share the identical relation model and seam. The difference is purely risk surface.


## Decision

Adopt Design 1 (Sidecar Relations Index) as the default, with two grafts from Design 2.

RELATION MODEL. A relation is an ordered (from: GroupId, to: GroupId, RelKind) with a bounded signed coefficient, encoded as a generalized-Lotka-Volterra-style community matrix A over species/lineages where A[i][j] = signed effect of group j on group i. The sign pattern names the kind: mutualism/synergy (A[i][j]>0, A[j][i]>0), competition (both <0), parasitism/predation (mixed: consumer >0, host/prey <0). The continuous LV ODE is NOT integrated (no transcendentals/stiffness); each generation A collapses to a per-group strictly-positive multiplicative fitness factor — the soil/climate precedent. Modeled over an opaque GroupId so it slots onto whatever R3/ADR-013 lands; relations are group-on-group, never per-organism (inv #6).

EMBEDDING. A pure, deterministic embed(&Genome, &Phenotype) -> [u16; D] computed in-core per SPECIES/lineage representative (founder genome or quantized member centroid — never per organism). It walks genome.loci → parameters → tags in their existing stable Vec order (no HashMap, no transcendentals). Lean default block layout: [phenotype 5 dims = gp.rs Trait::ALL express() values, already ordered/[0,1]] ++ optional later blocks (bounded-K first-K parameter as_unit_scalar() zero-padded/truncated; integer-modulo ontology fingerprint over SO/GO ids, M buckets) APPENDED so existing index rows stay valid. Every f64 is quantized to u16 = round(x.clamp(0,1)*65535) before it can touch the hash or be shipped. D, K, M, the u16 scale, and block order are PINNED in this ADR (start D=5 phenotype-only; grow by appending). embed()'s output is hash-neutral ONLY in the view-only slice (it rides the snapshot path); the moment the matrix couples into selection, embed() becomes hash-load-bearing and part of the re-pin.

VECTOR DB AT THE BOUNDARY. A FILE-BASED embedded index used as a write-then-read SIDECAR (default: sqlite-vec, loadable SQLite extension, dual Apache/MIT, single-file .db; LanceDB if columnar features wanted). The HARNESS writes per-species quantized embedding rows to a sidecar index file at epoch boundaries (off the sim hot path, like the existing --specimens export); the relations view opens it read-only for KNN/neighbourhood queries. The engine never links into sim-core or the game binary (inv #1); it lives behind a thin driver crate crates/relations-index built on the oracle-slim template ($RELDB_BIN → pinned path → PATH resolver, plain-data params in, Result<_, RelError> out, caller-derived seed via det::derive_seed, determinism judged downstream not by output bytes). The engine/extension version is pinned in DECISIONS.md (inv #7). Because the sim core does not even open the file, this honours inv #1 in letter and spirit — and sqlite-vec is permissive (no GPL concern for any candidate).

SEAM. A new trait RelationModifier { fn fitness_factor(&self, focal: GroupId, presence: &OrderedPresence, matrix: &RelationMatrix) -> f64 } mirroring EnvironmentModifier/ClimateModifier, returning a strictly-positive factor in [0.5,1.5] (ADR-005 no-extinction). Default impl: factor = clamp(1 + k * sum_j(A[focal][j] * presence_j), [0.5,1.5]), presence_j a bounded [0,1] abundance proxy summed in sorted GroupId order. The in-core matrix A is built by EXACT quantized cosine over the [u16;D] vectors in sorted GroupId order (multiply/add/clamp only) — the vector DB is NOT on the dynamics path at all. At inv-#6 cardinality (tens of species) O(K^2) exact cosine is trivially fast AND bit-reproducible.

GRAFTS FROM DESIGN 2 (cheap, safe): (1) the relations-view UI gains a dual panel + provenance badges — a deterministic/authoritative K×K signed-A heatmap in fixed GroupId order labelled "drives selection", and an advisory/approximate "nearest cousins" ANN overlay labelled "view-only, may differ run-to-run", which must never be conflated; plus graceful degradation (matrix+graph render fully if the index is absent; only the advisory overlay disappears). (2) SPECIFY but do not build the ANN→selection quantize+order firewall (sort by GroupId not float similarity; re-quantize similarity to u16 on a fixed grid; freeze per epoch; deterministically threshold the neighbour SET; STOP-THE-LINE on any raw-float read in selection) as the only legal future crossing, deferred behind a later ADR + scale trigger.



## Consequences

POSITIVE. Minimal determinism risk: the recommended first cut keeps the vector DB 100% view-only with ZERO crossing — the sim is byte-identical whether the DB is present, absent, or returns different neighbours run-to-run. Safety is STRUCTURAL (two physically separate computations: file index vs in-core matrix) not procedural (no hand-maintained firewall on the hot path). Faithful inv #1: the engine is a sidecar file the sim core never opens; the driver crate clones the gated oracle-slim pattern. Reuses everything (SPEC §2.2): express() as embedding, the soil/climate seam shape, the oracle-slim template, the --specimens off-hash export precedent. Full gameplay arcs (predator-prey boom-bust, host-coupled parasite drains, mutualism flywheels) come from the in-core signed matrix, NOT the DB — so Design 1 delivers them with none of the ANN risk. Slot-on-R3 ready (opaque GroupId). Cleanly sliceable: view-only first (no re-pin, no crossing), coupling later as ONE deliberate ledgered re-pin (the ADR-012 E3 discipline).

NEGATIVE / COSTS. (1) check_license.sh part-B is HARD-KEYED to the crate name "oracle-slim" and does NOT cover crates/relations-index — so "gated like oracle-slim" is unenforced until part-B is generalized to a boundary-crate list (or a sibling assertion is added). Part-A (GPL scan) covers any new crate automatically. This gate change is an acceptance criterion of the first slice. (2) The quantization contract (D, K, M, u16 scale, block order) is load-bearing and must be pinned here; changing it invalidates index rows and, post-coupling, forces a hash re-pin. (3) Hard dependency on R3/ADR-013 for GroupId to be meaningful — the dynamics half cannot fully land until multi-species exists (mitigated by the opaque-GroupId abstraction + shipping view-only first). (4) sqlite-vec brute-force KNN scales worse than HNSW — fine at PoC counts, a ceiling if lineage history grows large (the documented trigger to reconsider Design 2's server path). (5) Sidecar index files must be namespaced per run_id/work_dir or stale state misleads the view across runs (a hygiene, not determinism, concern since it is off-hash). (6) DOCUMENTATION HAZARD: "embed() is hash-neutral / provably cannot perturb the hash" is true ONLY for the view-only slice; once the matrix couples into selection, embed()'s output is read by hashed logic and is INSIDE the hash. The ADR text must scope that claim to the view-only slice so a future reviewer is not misled.

## Invariant risks

- inv #1 (process boundary): the vector engine must never link into sim-core or the game binary. RISK — check_license.sh part-B is hard-keyed to 'oracle-slim' and does NOT cover crates/relations-index; until Rel-1 generalizes it, the boundary is unenforced. The sidecar is a file the sim core never opens, so the spirit is honoured; the gate must be made to enforce it.
- inv #3 (determinism): the vector DB (ANN/HNSW float ordering) is non-deterministic and inv #7 pinning does NOT make it bit-stable. RISK — any raw ANN float (similarity score OR neighbour-set membership) read by selection() breaks bit-determinism SILENTLY and intermittently (passes local gate, fails cross-run/cross-build). The default design eliminates this by keeping the DB 100% view-only with zero crossing; the dynamics use exact in-core quantized cosine. The deferred ANN→selection firewall (specified, not built) is STOP-THE-LINE if ever opened.
- inv #3 (re-pin discipline): Rel-5 coupling is the single deliberate re-pin (next after ADR-012 E3 0x9fad_2c9f_d298_f73a). RISK — embed()'s 'hash-neutral' property holds ONLY in the view-only slices; at coupling it becomes hash-load-bearing. Must be scoped in docs so a reviewer is not misled.
- inv #5 (pluggable behind a trait): RelationModifier must mirror EnvironmentModifier/ClimateModifier exactly (strictly-positive multiplicative factor); swapping a service-backed impl must not touch sim-core arithmetic.
- inv #6 (species/region granularity): embeddings and relations are per-species/lineage, never per-organism; the per-individual factor is a lookup by the organism's GroupId.
- inv #7 (versions pinned): sqlite-vec engine/extension version pinned in DECISIONS.md; D/K/M/u16-scale/block-order pinned in this ADR (load-bearing for index + post-coupling hash compatibility).

5. Confirm REJECTING Design 2's Qdrant ANN→selection feedback for the PoC: it forces either relaxing the inv-#1 license gate (to link qdrant-client) or hand-rolling a wire protocol, adds server lifecycle surface, and routes an approximate signal back into the hash — for zero modeling gain at tens of species. Keep it specified-but-deferred behind a future ADR + a lineage-scale trigger?

# ADR-017: Layered evidence-based ecosystem — multi-fidelity coupling (fast deterministic sim ⇄ deep E. coli background), deep K-12 model, the earned-edit game mode, the third species, and the boundary vector-DB relations

> **DRAFT — `ecoli-layered-architecture-design` workflow (8 agents, web-researched real E. coli). DESIGN ONLY — awaiting human sign-off. Builds on ADR-013 + the multi-species prep plan.**

## Context

The vision stacks three coexisting layers on the existing engine: (L1) KEEP the fast 30 FPS bit-deterministic ChaCha8 Wright-Fisher core (ADR-013 CHEMOSTAT-J; sim-core/lib.rs:467 step, hash_world lib.rs:739 pinned 0xf795_eac4_112f_acd5) with the 9-trait genome and the JSON-roster multi-species generalization (multispecies-prep SpeciesSpec/R3); (L2) a DEEP, evidence-complete REAL E. coli K-12 MG1655 model running at a DIFFERENT fidelity, edited occasionally as an EARNED game action, whose ecosystem impact is computed SLOWLY in the background and fed back into the fast sim; (L3) a THIRD soil/gut species interacting with E. coli (decomposer / predator / mutualist), with soil nutrient becoming a function of nearby organisms (plants shed detritus → E. coli mineralizes detritus → free nutrient); plus a process-boundary VECTOR DB indexing all 3 specimens' embeddings to relate them (similarity / lineage / synergy / parasitism / predation).

The hard part is already solved by precedent. selection() (lib.rs:249, verified) is the SINGLE hashed dynamics system; its weight is a PRODUCT of strictly-positive [0.5,1.5]-bounded factors (base fitness × soil_mod × clim_mod) built in stable sorted-OrgId order over a BTreeMap (no HashMap), with exactly two RNG draws per offspring from the single SimRng. A species-level change already enters deterministically BETWEEN steps via with_genome_and_rng (lib.rs:642, verified — replace/restore RNG dance, re-expresses phenotype, updates BaseGrowthRate), and a player edit is already a journaled Action replayed bit-exact. ResourceField {light, free_nutrient, detritus} is inserted but unwired/hash-neutral. The crates/oracle-slim template (std-only, $BIN→pinned→PATH resolver, plain-data in / Result out, NO byte-compare of output, zero normal/build deps) is the proven inv-#1 subprocess pattern. The fixed.rs q(x)=round(clamp(x,0,1)*65535)→u16 quantization contract (F-1 landed) is the single audited float→integer chokepoint. So ~70% of the layered vision's PLUMBING already exists; the net-new work is the multi-fidelity firewall, the deep E. coli data artifact, the earned-edit economy, and the third species/vector-DB — none of which require new hash machinery, only new journaled Actions + an offline-baked quantized table + ordered modifier seams.

The defining new constraint is inv #3 applied to a LOAD-BEARING feedback (unlike the view-only Rel/vector-DB firewall): an E. coli edit MUST change the ecosystem, yet FBA/MOMA are double-precision LP/QP solves that are NOT bit-reproducible across solver/platform/threads (GLPK ships an exact-rational simplex precisely for this reason), so the deep compute can never run on the hash path and its result must re-enter only as a quantized, ordered, journaled integer committed at a deterministic boundary.


## Decision

Adopt the producer/consumer multi-fidelity firewall, reusing the existing seams verbatim. ONE direction of data flow; the slow non-deterministic deep compute is always the PRODUCER, the fast sim only ever CONSUMES a committed quantized integer.

1) DEEP E. coli AT THE BOUNDARY. New crate crates/oracle-fba, a structural clone of crates/oracle-slim (std-only, $FBA_BIN→pinned→PATH resolver, plain-data EditRequest in / quantized EditImpact scalars out, shells out to a COBRA/FBA CLI, links nothing — inv #1). It quantizes its output to u16 via the fixed.rs contract BEFORE returning; floats never escape the subprocess. PREREQUISITE (P0): generalize scripts/check_license.sh part B — currently hard-keyed to the literal "oracle-slim" (verified lines 58/63/65) — to a boundary-crate LIST [oracle-slim, relations-index, oracle-fba] with a per-crate zero-normal-dep assertion, ELSE the new boundary is mechanically unenforced.

2) PRECOMPUTE-TO-FROZEN-TABLE. The E. coli CORE model is ~134-137 genes, so the entire single-gene-KO landscape is ~134 LP solves. Bake it OFFLINE at asset-bake time into an inert, content-hashed, ORDERED Vec<KoImpact> indexed by LocusId (never HashMap): {delta_growth_q:u16, exchange_flux_q:[u16;K], essential:bool}. At runtime a single-gene edit is a DETERMINISTIC TABLE LOOKUP — the LP solver dependency never enters the game. Live boundary FBA is reserved for novel multi-gene combos / Layer-3 medium-dependent re-solves and returns through the same channel.

3) FEEDBACK INTO THE FAST SIM via the existing product-of-bounded-positive-factors seam. A new EcoliEditModifier returns a strictly-positive [0.5,1.5] factor derived from the committed delta_growth_q, looked up per organism by GroupId/SpeciesId in stable order, multiplied into selection()'s weight product — structurally identical to soil::EnvironmentModifier and climate::ClimateModifier. The committed integer is stored in a per-species BaseGrowthRate-shaped slot read BETWEEN steps via the with_genome_and_rng replace/restore dance, applied at a deterministic epoch boundary in SpeciesId order. Wiring a non-zero coefficient is a deliberate hash RE-PIN (new pinned literal at lib.rs:832 + ledger line + 🛑 human sign-off), kept selection-NEUTRAL (zero coefficient) on introduction so introduction is hash-neutral and only the activation re-pins (the climate-extremity precedent).

4) GAME MODE = OVERSIGHT, a third top-level mode (serde GameMode enum on the run config, serde-default to today's behavior) beside Sandbox and Mission/campaign. It REUSES the Mission predicate/objective spine but earns the edit budget at runtime: a RNG-free, core-side, hash-NEUTRAL score→credit accrual over the per-gen stats stream (lives in the harness/env layer like edits_used, recomputed on replay, adds ZERO bytes to hash_world). Two-tier edit economy: cheap FAST region edits (existing ApplyEdit/ApplyEditRegion) vs a rare, expensive, deep-compute-triggering ApplyEcoliEdit.

5) THIRD SPECIES = a DATA roster entry (data/species/<key>.json SpeciesSpec), not new machinery. Ship OPT-A (a Decomposer detritivore) first — it closes the Layer-3 detritus→free_nutrient cycle with the named convert tap already on the CHEMOSTAT-J roadmap and needs zero new interaction kernel (relation to E. coli is shared-pool competition). Then OPT-B (Bdellovibrio bacteriovorus predator) as the first F4 InteractionKernel, where an earned E. coli edit (KO ompF → predation-resistant) produces the most legible Layer-2 ecosystem swing. Relations are EMERGENT measured net-J flows (the in-core integer FlowMatrix built during trophic_transfer), NOT a fabricated input.

6) VECTOR DB = view-only analytics OVER measured flows. crates/relations-index (oracle-slim clone, sqlite-vec via a CLI-shim binary so it links nothing) indexes the in-core embed() RepVector [u16;D=9] (extended at F2 with an append-only metabolic-fingerprint block) and answers KNN nearest-cousins/lineage queries. The relations VIEW shows two provenance-badged panels: AUTHORITATIVE FlowMatrix heatmap (on-hash, drives dynamics) and ADVISORY ANN overlay (off-hash, may differ run-to-run). KNN output NEVER re-enters selection().

This SUPERSEDES the multispecies-prep Rel-5 (in-core cosine-into-selection) under CHEMOSTAT-J: relations-as-dynamics come for free from the conserved-J FlowMatrix, removing one re-pin; the cosine matrix is demoted to view-only embedding similarity.


## Multi-fidelity coupling (fast sim ⇄ deep E. coli)

FIVE-STAGE PRODUCER/CONSUMER PIPELINE, all reusing existing seams — no new hash machinery.

STAGE A — TRIGGER (in-core, RNG-free, hash-NEUTRAL). The player spends an earned credit, emitting a journaled Action::RequestEcoliEdit{ species:SpeciesId, locus:LocusId, edit_kind:KO|Knockdown|Activate, due_epoch:u32 }. locus is a real K-12 gene (genome::Locus with real CDS bases; crispr lands real edits on real codons). This only RECORDS the request + the epoch its result is due; it does not block the sim and does not itself alter selection.

STAGE B — DEEP COMPUTE (boundary subprocess, OFF-HASH, async, non-deterministic-tolerant). crates/oracle-fba shells out to the COBRA/FBA CLI: edit → Boolean GPR rules → reaction flux bounds → FBA (or MOMA for KO realism) on the medium → read-outs {growth ratio mu/mu_wt, key exchange fluxes (glucose/O2 uptake, acetate/CO2/NH4+ secretion = the Layer-3 decomposition rates), essential flag}. For single-gene edits this is the precomputed frozen-table lookup; live solves only for novel combos.

STAGE C — QUANTIZE + FREEZE (the firewall). The float result NEVER crosses into hashed logic. The boundary emits integers only via fixed::to_unit_u16: growth_ratio_q:u16 + an ordered Vec<(ExchangeId:u16, delta_q:i16)>. The whole struct is content-hashed (over the quantized BYTES, not the floats). Past this point only quantized integers exist, so the result is bit-portable even though the FBA that produced it was not.

STAGE D — BUFFER + SCHEDULED COMMIT (kills wall-clock leakage — the async-determinism crux). The boundary result lands at an unpredictable wall-clock time, so it is NEVER applied on completion. When it arrives, the harness writes a SECOND journaled Action::CommitEcoliImpact{...content_hash...} tagged with the due_epoch. The core applies it at the START of that epoch boundary, in ascending (SpeciesId, req_id) order. If the deep job has not finished by due_epoch, the commit deterministically SLIPS to the next epoch (the slip itself is a journaled fact), so solver speed cannot change the result OR its timing. On REPLAY, CommitEcoliImpact is read straight from actions.ndjson — the deep compute is NEVER re-run; the integer is consumed directly. This is the ApplyEdit replay pattern extended to a producer/consumer split.

STAGE E — APPLY INTO THE FAST SIM (in-core, ordered, the deliberate re-pin). At the scheduled boundary the committed impact feeds back through two existing seams: (1) growth_ratio_q becomes one more strictly-positive [0.5,1.5] factor in selection()'s product via EcoliEditModifier, looked up by GroupId in stable order; (2) the signed exchange_deltas become an ordered ResourceField tap at the epoch boundary (detritus→free_nutrient mineralization for Layer 3), routed through the Ledger's named taps so ledger_closes holds. The content_hash is folded into hash_world (lib.rs:739) so a different committed impact is a different ledgered run.

DETERMINISM BOUNDARY, explicit. INSIDE the hash: only QUANTIZED integers committed at a tick/epoch boundary via a journaled Action (deterministic inputs, exactly like a player ApplyEdit); the per-species committed slot read between steps; the ResourceField epoch taps; the content_hash. OUTSIDE the hash: the FBA/MOMA LP/QP solve itself, all raw fluxes, the COBRA subprocess, the sqlite-vec ANN, the wall-clock arrival time, any float before fixed::to_unit_u16. The crossing is one-way and integer-only. ACCEPTANCE TEST (the firewall proof, a CI gate): the sim hash is byte-identical whether oracle-fba/the sidecar is present, absent, or returns DIFFERENT bytes run-to-run — UNTIL the operator deliberately commits a quantized value, which is then a recorded, replayable input; replay never re-runs the deep compute. Wall-clock independence is enforced by due_epoch scheduling + journaled slip, tested with artificial solver-latency injection.


## Deep E. coli model

GENOME DATA SOURCE. Real E. coli K-12 MG1655: CDS bases from GenBank U00096.3 / RefSeq NC_000913.3 (ASM584v2) — US-government-produced, PUBLIC DOMAIN, clean to vendor. Metabolic model: the E. coli CORE model (134-137 genes / 95 reactions / 72 metabolites of central carbon metabolism — glycolysis, PPP, TCA, oxidative phosphorylation, fermentation, key exchanges), as the first deep substrate; iML1515 (1515 genes / 2719 reactions / 1192 metabolites) is the same data shape and swaps in later behind the same trait (inv #5). GPR Boolean rules (gene→reaction; OR=isozymes, AND=complex subunits) and RegulonDB v12 TF→gene edges for Ring 2. Annotation cross-ref from EcoCyc v29.

STOP-THE-LINE LICENSING GATE (must be resolved by the human BEFORE any model file is vendored): the BiGG SBML models (e_coli_core AND iML1515) carry a UC San Diego clause ("commercial purposes should contact UCSD Technology Transfer") that the BiGG site's CC BY 4.0 database statement CONTRADICTS — this directly threatens the project's stated inv-#1 commercial-freedom rationale. The boundary protects against GPL CODE linking, NOT against restrictive DATA terms. The real CDS bases are clean regardless; the metabolic model + GPR may need re-derivation from a permissive source. EcoCyc carries registration/redistribution terms; RegulonDB is generally permissive. Pin the chosen GEM release, the COBRA/FBA solver+version, the K-12 annotation, the quantization contract (UNIT_SCALE=65535, K_EXCHANGE, block order), and sqlite-vec in DECISIONS.md (inv #7).

THE BACKGROUND MODEL (how an edit's ecosystem impact is computed). The COBRA chain: edit → gene state → Boolean GPR → reaction flux bounds → FBA (a linear program: maximize the biomass objective subject to steady-state S·v=0 and per-reaction bounds on a defined medium) → read-outs {growth-rate mu, exchange fluxes, essentiality}. Use MOMA (a QP finding the nearest feasible flux to wild-type) for KO realism. Validated science: the core/iML1515 lineage reproduces the KEIO single-KO essentiality collection at ~93% accuracy. KEY MOVE: precompute the ~134-solve single-KO landscape offline → quantized frozen ordered table → runtime is a deterministic lookup.

SCOPE / "optimally complete". Evidence-complete for ONE coherent subsystem (central carbon metabolism), NOT the whole 4.64 Mbp / ~4,500 genes (the y-ome shows ~34-35% of genes still lack experimental function, so 'complete' is bounded by biology). 134 loci is ~15x the current 9-param toy genome but trivially inside the ordered-Vec genome model and inv-#6 species granularity. THREE-RING ONION (ship the inner ring; outer rings are pure data growth): RING 0 sequence/annotation — one genome::Locus per gene (name=real symbol "pfkA"/"gltA"/"ppc", sequence=real CDS bases validated by DnaSequence, OntologyTags.so_term=SO:0000704 gene, go_refs=real GO MF terms); RING 1 metabolic cross-reference — a sibling table keyed by LocusId {bigg_reaction_ids, gpr_clause_index, essentiality_q} (DATA, not code, inv #2); RING 2 network — the GPR Boolean rules + RegulonDB edges as an ordered adjacency table (makes an edit non-local and is what the vector DB embeds). HARD SEQUENCING: a STRUCTURALLY-distinct 134-locus genome cannot EXPRESS correctly until F2-ontology-rekey lands (gp.rs:97-116 hardwires the 9 traits to flat param indices 0..8, so a non-canonical layout deterministically MIS-expresses); F2 is gated behind the CHEMOSTAT-J keystone re-pins + 🛑 sign-off. Layer 2 is NOT evidence-complete until expression is ontology-keyed.


## Game-mode loop (Oversight)

OVERSIGHT mode (serde GameMode enum on the run config, serde-default to today's Mission/Sandbox behavior — one DATA toggle, not a code fork), a third top-level mode beside SANDBOX (no scoring/credit, both edit tiers free) and MISSION (campaign.rs grader, fixed pre-paid edit_budget). OVERSIGHT = Mission's Scenario+predicate spine, but the budget is EARNED at runtime.

EARN POINTS (score→credit, RNG-free, core-side, HASH-NEUTRAL). Credit is an integer accrued from ecosystem-improvement signals the predicate engine already produces: CREDIT += clamp(quantized(objective_progress_delta_this_gen), 0, PER_GEN_CAP) — you earn for moving a region's allele/biomass/joule-pool/FlowMatrix-health toward the objective band, capped per generation so grinding is bounded. The accrual is a pure function of the per-gen stats vector, so it adds ZERO bytes to hash_world (the Sourced/predicate-engine path rides the snapshot/stats stream, never the RNG/hash path). Credit is harness/env state (like edits_used in ScenarioResult), recomputed deterministically on replay from the same stats stream — reproducible WITHOUT being world-hash state.

EARN AN E. coli EDIT (two-tier spend). When OVERSIGHT-credit ≥ ECOLI_EDIT_COST the player may queue ONE deep edit (gated like _can_spend_edit; over-budget/under-credit edits refused). Cheap FAST region edits (existing ApplyEdit/ApplyEditRegion) are the frequent immediate verb; the rare expensive ApplyEcoliEdit is the deep-compute-triggering verb, a SPECIES/REGION-level operator action (inv #6, no per-organism surgery) targeting a real K-12 locus.

BACKGROUND IMPACT. Dispatches to oracle-fba (Stages B-D above): for single-gene edits the deterministic frozen-table lookup; the result is buffered and committed at the next epoch boundary as a journaled CommitEcoliImpact, applied via EcoliEditModifier + the exchange-flux ResourceField tap. The sim NEVER blocks while computing (inv #5 in-core fallback). Layer-3 closure: the SAME quantized exchange fluxes (NH4+/acetate secretion) ARE the decomposition rate, so an E. coli edit reshapes the detritus→free_nutrient mineralization the plants grow in.

FEEDBACK (renderer read-only, inv #2). The view shows: a CREDIT meter; an "E. coli edit unlocked" prompt at threshold; a deep-edit workbench (real gene names + GO tags + the precomputed impact PREVIEW from the core-exported table — renderer requests, core applies); a "computing in background…" state (sim keeps playing); and on commit an INSPECT diff showing the species' growth/secretion/soil-flux change with Sourced provenance (which gene, which FBA model version). GDScript computes NO biology.

The game-design tuning knobs (PER_GEN_CAP, ECOLI_EDIT_COST, score curve, reversibility, epoch cadence) are Assumption-class — wrapped in Sourced<T> flagged Assumption and given an explicit DECISIONS.md allowlist entry distinct from biological constants. This is its OWN ADR (ADD-4) on the sci-game Tier 0/1 spine; recommend deep edits PERMANENT for the PoC (edits have consequences) and a FIXED epoch cadence (cleaner determinism).


## Third species + vector DB

THIRD SPECIES = a DATA roster entry (data/species/<key>.json SpeciesSpec via the validating Rust builder — the only JSON→Genome path, inv #2), not new machinery, because the CHEMOSTAT-J substrate already conserves J across all species and carries a TrophicRole{Autotroph,Heterotroph,Mixotroph,Decomposer}. Three evidence-grounded candidates, each a real soil/gut microbe with a published genome-scale metabolic model so oracle-fba scores its edits on the SAME footing as E. coli:
- OPT-A (RECOMMENDED FIRST) Decomposer detritivore: closes the Layer-3 cycle with ZERO new kernel — plants(autotroph) shed detritus on death (the ledger death-tap already routes residual J→cell detritus) → the decomposer's named convert tap turns detritus→free_nutrient → plants re-uptake. Relation to E. coli is shared-pool COMPETITION (already real in F3 contention). Grounded in soil C/N mineralization science.
- OPT-B (headline gameplay) Bdellovibrio bacteriovorus, role=Heterotroph(predatory): a real rhizosphere predator that invades/lyses gram-negative prey (recognizes E. coli OmpF porin), with a published GEM. Predation IS the F4 trophic_transfer kernel: net J flows prey→predator with an efficiency tax, accumulated into FlowMatrix A[pred][ecoli]>0, A[ecoli][pred]<0. An earned E. coli edit (KO ompF → predation-resistant) is the most legible Layer-2 swing. Introduces the first InteractionKernel — its prey-contention/visibility (frozen start-of-tick prey snapshot vs immediate ordered apply) MUST be pinned as a design decision, not discovered in impl.
- OPT-C Rhodopseudomonas palustris mutualist, role=Mixotroph: textbook E. coli↔R. palustris syntrophy (E. coli secretes acetate R. palustris consumes; R. palustris fixes N and cross-feeds NH4+) — bidirectional, A[i][j]>0 AND A[j][i]>0, needs two exchange-flux channels.
Recommendation: ship OPT-A first (pure role + convert-tap), then OPT-B (first kernel + headline edit impact).

VECTOR DB = view-only analytics OVER measured flows (its role INVERTS under CHEMOSTAT-J per ecology F6). Pipeline: (1) IN-CORE embed() — pure, ordered, HashMap-free, u16-quantized RepVector [u16;D=9] over gp::Trait::ALL (D=9 not the rel-draft's stale D=5; verified). For the 3-specimen relate goal, EXTEND embed() at F2 with an APPEND-ONLY metabolic-fingerprint block {role one-hot + budget[5] simplex + exchange-flux-sign fingerprint} so cosine similarity captures METABOLIC relatedness (complementary fluxes=synergy; shared limiting uptake=competition; predation affinity). Pin D, block layout, scale (65535), and block order in the ADR BEFORE any row is written (inv #7) — changing them invalidates all index rows. (2) AT THE BOUNDARY — crates/relations-index, an oracle-slim clone running sqlite-vec (single-file, dual Apache/MIT, brute-force KNN fine at inv-#6 cardinality) wrapped in a CLI-shim binary so the crate LINKS NOTHING (sqlite-vec's loadable-extension model would otherwise link — load-bearing for inv #1). The harness writes per-species RepVector rows at {reset, epoch, FORK} from a SEPARATE off-RNG GeneSimEnv. (3) THE VIEW — read-only KNN nearest-cousins + lineage neighbourhoods; a fork's child RepVector is near its parent by construction (small edit delta) so KNN clusters lineages. A stable LineageId assigned at reset and INHERITED by every fork descendant is the fork-stable id the DB needs. Dual-provenance view: AUTHORITATIVE signed FlowMatrix heatmap (on-hash measured J-flows) + ADVISORY ANN overlay (off-hash, may differ run-to-run) — never conflate.

THE FIREWALL (structural, not procedural): embed()'s u16[D] is the ONLY artifact crossing core→DB. The on-hash relation signal is the in-core integer FlowMatrix; the DB reads an EXPORTED COPY and its KNN output NEVER re-enters any hashed system. Acceptance test: sim hash byte-identical whether the sidecar is present, absent, or returns different neighbours run-to-run.


## Consequences

POSITIVE. The multi-fidelity coupling needs NO new hash machinery — it reuses the verified selection() product-of-bounded-positive-factors seam, the with_genome_and_rng between-steps mutation, the journaled-Action replay path, the fixed.rs quantization chokepoint, and the oracle-slim subprocess template. The async-determinism crux (wall-clock leakage) is solved by the due_epoch buffer + journaled slip + producer/consumer Action split, so solver speed cannot change the result or its timing, and replay never re-runs FBA. The precompute-to-frozen-table move collapses the entire FBA float-non-reproducibility hazard for single-gene edits into a deterministic lookup, removing any in-game solver dependency. Layer 3 is biologically real and nearly free: the FBA exchange fluxes ARE the decomposition rate, so the metabolic oracle and the soil pools couple through the SAME quantized numbers, and the third species is a roster entry. The CHEMOSTAT-J re-grounding REMOVES a re-pin (Rel-5 cosine-into-selection is superseded by the emergent measured FlowMatrix). Introduction is hash-neutral throughout (zero-coefficient modifiers, append-only embedding block, serde-default Actions); only deliberate activation re-pins.

NEGATIVE / COST. A tall STACKED re-pin chain: the load-bearing EcoliEditModifier wiring, the exchange-flux ResourceField tap, and the content_hash fold are each a deliberate hash re-pin layered ON TOP of the already-stacked CHEMOSTAT-J (6+) and R3/Rel re-pins — each one ledgered commit + new pinned literal + 🛑 sign-off, multi-week, sequenced. HARD SEQUENCING: the evidence-complete deep E. coli (and the metabolic embedding block, and distinct per-species role/budget/affinity) are BLOCKED behind F2-ontology-rekey, itself behind the CHEMOSTAT-J keystone re-pins + R3 multi-species (a stop-the-line inv-#6 gate). VALUE-only homogeneous-layout species and the earned-edit ECONOMY can prototype against a VALUE-only E. coli stand-in BEFORE F2; the evidence-complete deep E. coli cannot. Two STOP-THE-LINE gates must clear before any data is vendored: (1) the BiGG/UCSD non-commercial licensing contradiction vs the inv-#1 commercial-freedom rationale; (2) generalizing check_license.sh part B to a boundary-crate list, or the new boundaries are mechanically unenforced. The game-design tuning numbers are Assumption-class and need an explicit provenance allowlist. FBA and ANN are not bit-stable across versions — acceptable ONLY because both live off-hash and feed back only via the frozen, journaled, content-hashed artifact.

## Invariant risks
- INV #1 MECHANICALLY UNENFORCED until S0: scripts/check_license.sh part B is hard-keyed to the literal 'oracle-slim' (verified lines 58/63/65) and covers neither relations-index nor oracle-fba — a new boundary crate could accumulate a GPL/linked dep undetected. Generalize to a boundary-crate LIST FIRST. sqlite-vec's loadable-extension model would LINK an extension unless wrapped in a CLI-shim binary (the oracle-slim shells-out precedent) — honoring this is load-bearing.
- INV #1 DATA LICENSING (distinct from code linking, STOP-THE-LINE): the BiGG e_coli_core/iML1515 SBML carries a UCSD non-commercial clause that contradicts BiGG's CC BY 4.0 statement and threatens the project's commercial-freedom rationale; the boundary protects against GPL CODE, not restrictive DATA. EcoCyc carries registration terms. Real K-12 CDS (NC_000913.3, public domain) is clean. Human ruling required before any model file is vendored.
- INV #3 FBA FLOAT NON-REPRODUCIBILITY: LP/QP solvers are double-precision, non-associative, NOT bit-stable across solver/platform/threads (GLPK ships an exact-rational simplex for this reason). If any float exchange flux ever bypasses fixed::to_unit_u16 into selection(), or a live solve is consumed mid-tick, the hash divergence is SILENT and intermittent (passes local gate, fails cross-run/cross-build). Mitigation: precompute+quantize+freeze+journal; never read solver output live; fold table content-hash into the hash; the present/absent/different-bytes hash-identity CI gate.
- INV #3 ASYNC WALL-CLOCK LEAK (the subtlest, NET-NEW): applying the background result on completion instead of at the journaled due_epoch in SpeciesId order makes the hash depend on solver speed and breaks replay. The buffer-and-apply-at-next-epoch + journaled slip is MANDATORY and must be tested with artificial solver-latency injection.
- INV #3 RE-PIN STACKING: the load-bearing EcoliEditModifier wiring, the exchange-flux tap, and the content_hash fold are deliberate re-pins layered on top of the already-stacked CHEMOSTAT-J + R3/Rel re-pins; risk of a re-pin landing without the ledger line, or the committed value folded as a reset constant instead of the live per-species value at hash time. Keep selection-neutral on introduction; only activation re-pins.
- INV #3 ANN RE-ENTRY: any raw ANN distance/neighbour-set read by an on-hash system breaks determinism silently. The design eliminates this structurally (DB 100% view-only; on-hash relations are the in-core integer FlowMatrix), but a future 'use cousins to nudge dynamics' temptation is STOP-THE-LINE and must be refused or routed only through a deliberate quantized+ordered+journaled re-pin. Also: building the index on the stale D=5 embedding (corrected to D=9 over gp::Trait::ALL) would mis-lay every row; pin D/block/scale/order before the first write.
- INV #2 BIOLOGY IN CORE: genome→phenotype/Strategy expression, the deep→fast trait correspondence map, the KO-table lookup, and the FBA-delta application are biology and live in crates/genome + crates/sim-core / the boundary service — never GDScript.
